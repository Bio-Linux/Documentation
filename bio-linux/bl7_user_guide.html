    1. User Accounts

      • Administration accounts

        The user you create when installing the system will automatically be given administrative, aka sudo, privileges. This means that this user, by prepending the word "sudo" to commands, can execute those commands with root administrator privileges. To give others sudo (administrator) privileges for the whole system select Users Accounts from the System Settings menu and change the account type for that user to Administrator. For more fine-grained control, including restricting access to a specified set of commands, you will need to edit the sudoers file.

        Another thing to note about administrative users is that they are added to the ssh group by default. This means they can log into the system from other machines using ssh or NX.

        Older machines upgraded from Bio-Linux 6.0 to 7 may have a NEBC Admin User account with the username nebc. This is an account that NEBC staff could use (solely for supported users and with permission) to log in to the system remotely, in case of an emergency.  We no longer add this by default - you can safely delete this account.

      • Create new accounts

        To create a new account you will first need to log into an account with root (sudo) privileges. We recommend that you use the graphical tool for managing users and groups. To start this tool, go to the System Settings menu (red icon with cog and spanner) and choose: User Accounts. 

        Each user has a set of privileges, gained from being members of a set of particular groups. A new user created with the GUI will be created as a Standard User by default. The GUI in Bio-Linux 7 does not provide fine-grained control of user privileges.  To discover what groups a user named user1 is in, open a terminal and type:

        groups user1

        The default behaviour of the graphical tool mentioned above creates fully functional user accounts on Bio-Linux. To create a functional Standard user using the command line, you would need to run this command:

        sudo adduser --add_extra_groups user1

        To upgrade the user to an administrator, you can then do:

        sudo usermod -aG adm,sudo,ssh user1


        Note that by default, Standard users are  not added to the ssh group. If you want a standard user to be able to log into your Bio-Linux system from another machine, using ssh or NX, then you must add them to the ssh group. Administrative users are included in this group by default.

        sudo usermod -aG ssh user1

        For further information, please see the Ubuntu documentation on User Management.

      • Add users to groups

        There are many instances when you may wish to add users to groups. For example, perhaps particular people need full access to the ARB pt_server, which on Bio-Linux means they have to be members of the arb group. Or perhaps a group of people wish to work on a set of shared files - you can make these easily accessible by creating a group and adding the users involved to this group.

        Unfortunately, Ubuntu 12.04 does not have a satisfactory graphical interface to manage group membership.   You should therefore add users to groups using the command usermod as in the examples above.  As a further example, to add user1 to the arb group, so they can make full use of the ARB software, you would type:

        sudo usermod -aG arb user1
        There is no flag to remove groups, so to achieve this you have to run groups user1 to see the list of groups and then sudo usermod -G group1,group2,group3,... user1 to reset the groups list explicitly.

      • Supported shells

        Z Shell is the default shell within Bio-Linux.  We like it for superior tab completion, succinct loop syntax and supporting conventions from other shells, among other things. Our choice means that you may have to undertake customisations of your own, such as setting environment variables, if you choose to use another shell such as tcsh. Customisations like aliasing the remove command, rm, to rm -i, so that users are asked to confirm whether they really wish to remove a file, are done through the /etc/zsh/zshrc file which is controlled by the bio-linux-zsh package on Bio-Linux systems. This file also loads commands from /etc/zsh/zshrc.d/customrc.zsh which is the file you should customise to add system-wide settings and environment variables.  Users should add their personal settings to ~/.zshrc.
      • The BASH shell is the default for Ubuntu and is also supported on the system, but we suggest that users should only use BASH if they have a particular preference for it.  In order to ensure compatibility with Ubuntu technical documentation and system scripts, BASH is the default root shell - see below.

      • Changing passwords

        You can change the password for accounts by going to the System Settings icon and choosing:Users Accounts. Then click on the Unlock button at the top right, and after unlocking select the user you wish to change the password for in the left hand pane, and click on the Change text beside Password in the right hand pane.

        To change your own password does not require doing the unlock step but you will need to confirm your existing password in the next step.

        Changing your password on the command line is faster. Just open a terminal and type:

        passwd

        Entering your new password when prompted. Detailed information about password security can be found on Chris Ihm's page on choosing a secure password.

        If you are logged in as an administrator, you can reset the password for other users by using the command:

        sudo passwd username

        The Bio-Linux FAQ includes an item on what to do if you lose your administrative user password.

      • Root shell

        If you are coming to Bio-Linux from other Unix or Linux systems you may be wondering what happened to the root account.  By default, the root account has no valid password in Ubuntu, and thus also in Bio-Linux. This means you can't login as root directly or use the su command to become root.

        The use of sudo negates the need for a root password. You can get a root shell using the command sudo -i or alternatively sudo bash. Note that bash is the default root shell for compatibility reasons. 

      • Default umask

      The setting of umask controls the default permissions on new files.  The default umask in Bio-Linux used to be 077, as set in the files /etc/profile and /etc/login.defs.  It is now set to the Ubuntu default of 002, which is more appropriate for single-user machines.  A umask is an octal number which specifies which permissions bits will never be set for newly created files and directories.  In this case, the choice of 002 means that all users may see files you create and those in the same group may write to them.  If you are sharing the machine and don't want any other users to read your files, you probably want to protect your whole home directory by doing chmod 700 $HOME.  This blocks access to all files within the home dir regardless of any permission settings.  For more fine grained control you can set the umask with the umask command. The umask setting is inherited through the process tree just like an environment variable, so any user can put a line into ~/.zshrc to set the custom umask they want.  Meaningful umask values are:
      • 077 - only the owner has access
      • 027 - users in the same group can read files but not write
      • 022 - anyone can read, only the owner can write
      • 002 - as above, but users in the same group can also write (default)
      • 007 - owner or group members can read and write, others have no access

 

    1. System Configuration and Daily Maintenance

      • Configuration files in /etc

        • As a general rule, system wide settings are controlled via text files held within the /etc folder.  Often a graphical or command-line application is used to manipulate the settings, or the settings are written in by the Debconf package configuration system, but sometimes it is necessary or desirable to edit the files directly.  See the final section of this guide for more information on text editors, but in general if you need to edit a system configuration file the invocation is:

        sudo nano <filename>

# for example:

sudo nano /etc/zsh/zshrc.d/customrc.zsh

        Here, the sudo command gives you permission to modify the protected system file, and nano is a simple text file editor.

      • Entering or changing network settings

        DHCP
        If your local network uses DHCP the Roaming network mode should detect your network without further action on your part.

        Static IP addresses
        If you have a static IP address you will need to ask your IT people for an IP address hostname, domain, gateway and DNS addresses.

        To configure your network settings open System Settings --> Network. To modify the settings for any of your network devices you will need to be working in an account with sudo privileges. 

        To set the address for your wired ethernet connection click on the Wired  connection and then the Options button.  Click on the IPv4 Settings tab. From the drop-down Method box, choose Manual. Then click on the Add button. Add your IP address, netmask, gateway, along with your DNS Server addresses and search domains (if desired) below. The search domains are the default domains in which your system will search any host when no domain is specified. This means if you have a machine called mymachinename.mysite.com, and you've entered mysite.com into the search domains, you can refer to the machine as just mymachinename, rather than writing out the full mymachinename.mysite.com each time.

        After you apply your changes, you should be connected to the internet. You can see this by looking for two small white, opposing arrows towards the right side of the top task bar. If you aren't connected, then you'll probably see a little signal icon (like a seashell, probably with an exclamation point in red on it) in the top taskbar. If you click on that icon, and choose the name of the connection you've just created from the list, a connection should then be established. 

        Further information on setting up your network can be found on the Ubuntu documentation pages.

      • Software updates

        Every night, your system will check for updated packages in the Bio-Linux bioinformatics software repository, the Ubuntu updates repository and the Ubuntu security repository.

        Packages that need updating from the Bio-Linux  bioinformatics software repository and the Ubuntu security repository are downloaded and installed automatically. For other packages, you will be prompted to use the Update Manager tool for this when you log in as an administrator.  If you are running a server that is in regular use but do not regularly log into the graphical desktop via an administrative account you should make time weekly to install new updates.

        By default, all package files downloaded to the system are kept in the cache /var/cache/apt/archives for a week and are then deleted.  After a big update you can manually clear out the cache to save space:  sudo apt-get clean

        For more information about installing, removing and updating software including changing the Bio-Linux default settings, and alternative software available for managing packages, please see our Package Management page.

        Working behind a proxy

        If your site uses an HTTP proxy for internet connections you will need to configure Apt, the package management system, to use the same proxy. There are some reports that there are problems using the Ubuntu Software Centre from behind a proxy, so you may need to use command line tools or the graphical tool Synaptic for your package management if you have problems with the Software Centre.

        You can set up Apt to use your proxy by creating a file called 01proxyconf  in the directory /etc/apt/apt.conf.d/  containing the following lines:

        Acquire { http::Proxy "http://<proxyhost>:<port>"; };

        For example, if your website was called somewhere.ac.uk, and the port was 2067, you would enter:

        Acquire { http::Proxy "http://somewhere.ac.uk:2067"; };

      • System administrator role

        A system administrator is responsible for the security of the machine, installation of software, creating new user accounts and performing backups. The system administrator is also responsible for ensuring that only people deemed trusted for identified tasks have the appropriate permissions and privileges (e.g. sudo privileges) to carry out those tasks.

        We recommend that you maintain a logbook for your machine. (We find a real, paper, hardcopy logbook is the most effective.) In this logbook, write down every change that you make. These notes should be kept clear and complete so that you can reconfigure the machine from a fresh installation should you experience a hardware failure and need to reinstall. Keeping such a detailed record of what you have done to your system may seem tedious and unnecessary until the day something goes wrong for the second time. Then, those comprehensive notes from the first time you solved your problem will come in mighty handy and save a lot of time.

        You should check that the system updates are being made correctly by looking at the package status reports (for example, through the Synaptic tool or by reading the file /var/log/cron-apt/log ). You should also check the security logs. Please see the section below for more on this topic.

      • System logs

        The system log files are located in the directory /var/log/  and contain records of activity on your computer including kernel version, drivers in use, email activity, webserver access, logins and attempted logins, and much more. While we would advise all systems administrators to be aware of the logs and to know how to check them manually, there is a program called logcheck installed on Bio-Linux that will help to automate this task. Documentation about logcheck can be found by reading the man page using the command:

        man logcheck

        Logcheck scans all the log files listed in the file /etc/logcheck/logcheck.logfiles  and using information in other files under /etc/logcheck  it "decides" which information to report to you. Essentially, logcheck reports everything that it is not configured to ignore.

        On Bio-Linux 6.0, logcheck is run daily at 6:02am, and the reports are sent to the mail spool for the primary administrator user (the user account you create during installation). This means the logcheck reports can be read by the system administrator using an email tool such as mutt or Evolution.

        However, it is likely to be simpler if you send the reports to an email account you check regularly. To do this, you need to edit  the /etc/logcheck/logcheck.conf file and set the SENDMAILTO option to your normal e-mail address. You can include multiple addresses to send the reports to, by listing the addresses separated by a comma.  To send the mail to an address outside your Bio-Linux system, you will need to configure the Exim mail program so it knows about your local SMTP server.  Please see the section Email on Bio-Linux.

      • Monitoring processes

        After you click on a button or icon to start up a program, there may be a short delay for some programs before you see them or can interact with them. Some programs show you an icon indicating that something is happening. However, there are a few that don't.  If after a few seconds, nothing appears to be happening, you can investigate whether the program has started or not.

        The easiest way to do this is to run the System Monitor tool - search "system" in the Dash Home. A list of what your system is running can be found under the Processes tab. If you go to the View menu, you can choose whether to view just the processes owned by the user you are logged in as, or all processes.

        There is also a command line tool you can use to see all the jobs a particular user is running on the system:

        ps -fu user1

        To find out what you yourself are running, you can type:

        ps -fu $USER

        You can replace $USER with your username in the above command, or just let the shell do this: it will replace $USER with your login name.  Note that ps -f shows only processes in the current terminal.  ps -fux gives a full listing and shows process relationships so this is a particularly useful invocation.


        To read about other options available with the  ps  command, type:

        man ps

        To list processes in order of the demand they are putting on your system (CPU usage), you can use the  top  program. Simply type:

        top

        on the command line. (press q to quit)

      • Backups

        Bio-Linux can be configured to use a second physical disk for automated backups using a script. The first administration account created on the machine will have an icon on the desktop that can aid in setting up a second drive or partition for automated backups. Information on backing up data can be found on our Backing up and Restoring page.

      • Firewall

        Bio-Linux comes preconfigured with the firewall program ufw. By default this is configured to allow incoming ssh connections only. This can be easily reconfigured using the graphical interface to ufw. Search for firewall in the Dash Home.

      • Enable and disable services

        Services are programs which run in the background on the machine. Examples include sshd, apache2, mysql, ufw, and crond. Such services run all the time when the machine is on, no matter who is logged in.

        We advise you not to turn off any of the currently running services as some tools expect them to be working, and recommend that you liaise with your IT department before you turn any new services on.  The Bio-Linux Live system runs a more limited set of services than an installed Bio-Linux system: on live systems, sshd, apache2, postgresql and mysql all inactive by default, whereas they are active on the installed system.

        You can start, stop or reload the configuration file for services from the command line. Eg.

        To de-activate the Apache web server from a terminal, type:

        sudo /etc/init.d/apache2 stop

        To tell PostgreSQL to reload the access configuration file:

        sudo pg_ctlcluster 9.1 main reload

        To restart ssh:

        sudo /etc/init.d/ssh restart

        The new Ubuntu upstart system is now running the show for several packages. Upstart is an event-based init daemon, taking the place of the older dependency-based init daemon. The upshot of it is that many of the scripts that control the startup and shutdown of the system are actually held in the /etc/init folder. You call them, as well as other init scripts from the /etc/init.d folder. That is: some of the files in /etc/init.d are scripts, but some are just softlinks to a file called /lib/init/upstart-job, which causes a script in /etc/init to be run (if the appropriate script is found there).  Frankly though, for most people, it makes no difference if nothing in the preceding paragraph makes any sense to you.

    1. Accessing and Connecting to Bio-Linux

      • Two ways to login remotely

        Two methods of logging into your Bio-Linux system from another machine are mentioned briefly here. We also provide a webpage with further details about accessing Bio-Linux, as well as a page of different ways you can run Bio-Linux.

        NX -   This is our recommended method of logging in remotely to Bio-Linux. This allows you to view the full desktop just as you would when sitting in front of the computer. Interaction can be very fast even over a modest internet connection.  To do this, you just need to install NX client software on the machine you want to connect to Bio-Linux from. Such client software is available for Windows, Linux and Macs. For further instructions, please see the Accessing Bio-Linux webpage.

        SSH client -          There are many pieces of client software you can use on other machines to connect to your Bio-Linux system using ssh. To connect from a Windows computer we recommend you use "PuTTY" which can be downloaded from here. If you wish to use Putty with Exceed you will need to enable "X11 forwarding" in the PuTTY preferences. You can also try the openssh client through Cygwin. In general, if you want a graphical connection, we recommend that you consider using NX.

      • Email

        The standard email client on Bio-Linux is  Evolution (which works much like Microsoft Outlook) and can be set up by clicking on the icon of an envelope on the right hand side of the top taskbar and choosing Set Up Mail... You will need to configure Evolution with the details of your incoming and outgoing email servers and indicate whether the POP3 or IMAP protocol should be used. You may need to speak to your local IT staff to determine the correct settings.

        If you want to set up Evolution to read the logcheck messages (described above), you need to be logged in as an administrative user. Then:

          1. Launch Evolution;

          2. Go to Edit --> Preferences;

          3. Click the Add  button;

            You will then be guided through a Mail Configuration wizard;

          4. On the Identity menu:

            • Full Name: "system mail" (or whatever name you find suitable for this account)
            • Email Address: someuser@localhost (provide your user name before the @ symbol)

            You don't need to fill in any further details on this menu;

          5. On the Receiving E-mail menu, you need to provide the following information:

            • Server Type: Standard Unix mbox spool file
            • Path: /var/mail/manager 

          6. Set the Server Type on the Sending E-Mail menu to Sendmail. That is all you should need to configure in Evolution.


          If you want to send system mail to an email address outside the Bio-Linux machine, you will need to reconfigure exim4, the default Message Transfer Agent (MTA) in Bio-Linux 7. You can reconfigure it by typing (in a terminal):

          sudo dpkg-reconfigure exim4-config
          If you're having problems configuring this, you can have a look at a the well documented Ubuntu documentation pages.

      • Copying files between computers

        Files available via a local or remote web or ftp server can be fetched onto your Bio-Linux system using a number of tools. For example, you can use a web browser,  you can use the Connect to Server... option under the Places menu in the top taskbar, or you can use the versatile wget  utility on the command line. There is also a standard command line ftp client for uploading files to an ftp site. Both wget and ftp are well described on the respective man pages, which you can view by typing:

        man wget
        man ftp

        For moving files securely between machines on which you have accounts, we recommend the use of  scp. This is part of the  ssh  package. It can be used to move files to or from any machine to which you have ssh access. So if you would normally log into a machine with the command:

        ssh user2@comp2

        then you can put a file from the current directory on machine comp1 onto comp2 using

        scp ./myfile user2@comp2:/home/user2/myfile

        You will be prompted for the user2 password. Then the file will be copied into the user2 home directory. See the scp man page for more details and examples.  This works the other way round too, copying from remote to local.

        There is also an 'sftp' command which emulates an ftp session over the ssh connection, and a powerful command called 'rsync' is useful for copying whole directories of files recursively. Again, the man pages for these commands are good first port of call for further details.

        If you want to transfer files to (or from) your Linux machine from a Windows PC, the free application WinSCP will do the job. If using a Mac, you can use the command-line scp or a tool such as Cyberduck.

         

      • Databases

        Bio-Linux comes with Postgres and MySQL databases preconfigured. On the installed system, these services are running by default. (On the Live system, they are not.) 

        If you wish to connect to either database system from another machine, you will need to reconfigure the Bio-Linux firewall to allow incoming connections on port 5432 (for Postgres) or 3306 (for MySQL). To modify the firewall settings use the Administration --> Firewall  option in the System menu.

        Futher details on configuring databases can be found on the SQL Databases on Bio-Linux page.

    2. Software Applications

      • Bioinformatics software

        Bio-Linux comes with a comprehensive bioinformatics documentation system. It provides access to categorised information about all the bioinformatics software on your Bio-Linux system, as well as some additional software that can be easily installed. On each user's desktop, there is a Bioinformatics Docs icon that can be clicked on to open up the documentation in a web browser. Alternatively, you can type the following URL into a webbrowser on your Bio-Linux machine:

        http://localhost/bioinformatics

        Most software described in those pages is pre-installed on your system; the Description section for a given piece of software will state explicitly if the software is not already installed, and will provide instructions on how to install it.

        If you want to find out definitively if a particular piece of software is installed and ready to run on your system, try the command

        which software-name

        Note that this will only find the name of the actual command, not software packages. (e.g. it will find blastp, but not blast+).

        You can search for installed and available software tools using the system package management tools.

        Some bioinformatics software can be run from options under the Applications --> Bioinformatics  menu on the desktop. All bioinformatics software, including software with graphical interfaces, can be started from the command line. Please refer to our Introduction to Bio-Linux tutorial for examples of working with the bioinformatics software on Bio-Linux.

      • Installing new software

        For a simple method to search for and install software, we recommend the new Ubuntu Software Centre.  Another graphical tool you will come across, which is also very good though slightly (not much) more technical is Synaptic.  You will find the Ubuntu Software Centre in the main Dash menu bar.  Synpatic can be started from the Dash Home.

        Your Bio-Linux system is pre-configured to look for software in a number of repositories. You can see which these are by going to the Edit menu of the Ubuntu Software Centre and choosing Software Sources...

        Please refer to the Bio-Linux Package Management page for further information about the default setup for Bio-Linux package updates and additional information about software management on Bio-Linux. Ubuntu provides provides online documentation about Adding and removing packages using graphical tools, while the Ubuntu Guide provides a useful outline of the command line you can use for installing and managing packages on your system.

      • Linux alternatives to Windows software

        Check out the Linux Alternative Project page.

      • Running or replacing Windows programs on Linux

        For many people, there are free, open source programs on Linux that have the same functionality as programs on Windows. Sometimes the programs have the same names, but often they don't. A categorised listing of Linux and Windows program names is on the Libervis Wiki.

        For further information on Linux alternatives to Windows programs take a look at http://www.linux.ie/newusers/alternatives.php


        There is also software, available via the Ubuntu Software Centre, that allows you to run much Windows software on Linux such as PlayOnLinux, which is based on Wine.

        One of the most common things people want to do is to run office software - document editors like Word, or spreadsheet software like Excel. The powerful OpenOffice suite of software comes as standard on your Bio-Linux system and can be found in the main Dash menu bar.  You may also want to install the Gnumeric Spreadsheet software, which is not on the default image but can be found via the Software Centre

        While OpenOffice does a lot, there are certain things, generally of interest to power users, that you can't do. VBA programming for example. In cases like this, or any software that there is no equivalent in Linux for, you could consider one of several options:

        1. Run emulation software like Wine (or PlayOnLinux, mentioned above), which then allows you to run Microsoft Windows software inside the emulator.
        2. Run Windows as a virtual system on your Linux system, by installing a product like those released by VMWare or VirtualBox on Linux and installing Windows to run virtually.
        3. Run Linux as a virtual system on your Windows machine by installing a product like those released by VMWare or VirtualBox on Windows and installing Linux to run virtually.
        4. Install a dual-boot system, allowing you to log into either Windows or Linux on your machine.


    3. Hardware

      Data may be stored in various physical storage devices, like CD-ROMs, DVDs, USB memory sticks and hard disk drives. These storage devices are attached into the global filesystem heirarchy on your system before the data on them can be accessed. The process of attaching the devices to your system is called mounting. The directory where the device is attached is called a mount point.

      Luckily, in most cases, your Bio-Linux system will detect and automount new physical storage devices under the /media area. Usually, it will also show you an icon on the desktop for filesystems it has mounted - like your underlying hard drive when you are working on the live system, or an attached USB key.

      Knowing about mounting and unmounting is still useful. For example, sometimes you will want to carry out more complex tasks like making files on your Bio-Linux system available to other machines, adding an extra hard disk into the regular filesystem hierarchy, or force-unmounting an unresponsive USB drive instead of having to reboot to deal with it.

      Some general information about file systems and mounting are available from the Ubuntu Community Help pages. See also the links at the bottom of that page.

      To list all the mounted devices on your machine, with the mountpoints, you can just type:

      mount

       

      • Removable devices - USB sticks and hard drives

        When you plug a USB key or removable device into your Bio-Linux computer it will appear as an icon on your desktop. Before removing an external storage device you should remember to unmount the device first. This can be done by right-clicking on the desktop icon and selecting Unmount Volume.

        Note: Users have reported seeing the error "Cannot mount volume - invalid mount option". It seems that this can be fixed by looking in /etc/fstab for a line something like "/dev/sdc1 /media/cdrom0 ...etc..." and commenting it out or deleting it entirely.
        Further details about mounting and unmounting USB sticks can be found in the Ubuntu Community Help pages.

      • Printing

        The printing administration interface can be found under System Settings --> Printing. This interface provides an alternative view to the web based interface of Cups which you may be familiar with from earlier versions of Bio-Linux. If you prefer, the Cups administration page is still available at:

        http://localhost:631

        You will hopefully find the Printing administration wizard easy to use but for further instructions on printer setup we would recommend you consult the Ubuntu documentation.

      • NFS mount filesystems

        You can make the drives of your Bio-Linux machine available to other machines at your site. For example, perhaps you have blast databases on one machine and would like to make these available to people working on other Linux systems. For this example, we assume your blast databases are in the folder: /home/db/blastdb

        The main steps for doing this are:

        1. Edit the file on the machine you want to export the Filesystem from -  e.g. here, this is the machine with the blast databases on it. Depending on the exact details of what you want to do, you could add a line to the file /etc/exportfs,  like:

          /home/db/blastdb mymachine.ip.add.ress(rw)

          which would give the machine with address mymachine.ip.add.ress the ability to mount the file system found at:

          /home/db/blastdb

          on this machine, and users on mymachine.ip.add.ress with appropriate permissions the right to read it and to write to it.

          For sharing a directory with a range of machines on the network, and with more fine grained control over read and write access you may use:

          /home/db/blastdb *my.domain.here(ro) my.ip.add.ress(rw)

          which would grant read-only (ro) access to all machines on the my.domain.here network and read-write (rw) access to my.ip.add.ress for the directory /home/db/blastdb. For more information on exports consult the man page:

          man exportfs

        2. Restart the nfs-kernel-server on the exporting machine -  You will need to restart the nfs-kernel-server on the machine on which you have just edited the /etc/exportfs  file. To do that, type:

          sudo /etc/init.d/nfs-kernel-server restart

        3. Edit the firewall settings -  On the machine you want to export the filesystem from (here, the one with the blast databases), go to Open Firewall Configuration. Then Edit --> Add Rule and select the advanced tab.  Choose "Allow" "In" "Both" and put the numeric address of the client machine in the From box.  Click Add.

        4. The above rule will open the firewall to all connections from that client.  It is possible to specify a range of hosts.

        5. Mount the Filesystem -  In the case of the export defined above, you would now be able to mount the filesystem on the second machine by typing something like the following on that machine:

          sudo mount mymachine.ip.add.res:/home/db/blastdb /my/blastdb/location

          If you wish the filesystem to always be mounted on your second machine at boot time, edit the /etc/fstab  file on this machine, and restart nfs-common:

          sudo /etc/init.d/nfs-common restart

          Please note -  If you want users to have write access on both machines, you will need to synchronize the user accounts across the machines as well so that all users have matching UIDs as revealed by id $USER.

      • Shared network drives

        You can configure your Linux machine to access Novel Netware shared drives on your site. The details of how to do this varies from site to site; you will need to know information about the drives at your site, as well as learning about how Linux uses mount points and the /etc/fstab  file.

        • One-off setup -

          You only need to carry out the steps in this section once to access a Netware share via IP from your Bio-Linux system.

          1) install the ncpfs package. If you are using command line tools, you could ty

          sudo apt-get update
          sudo apt-get install ncpfs

          2) create a mount point: this  is just a regular (empty) directory. We suggest that you put it  either in the user's home directory, or in the system-wide mnt folder:

          sudo mkdir /mnt/netware

        • Mounting the file system -

          This must also be re-done after any system reboot:
          ncpmount -S <remote server (logical) name> -A <remote server dns name>  -U  < remote system username> -u < local system username> -g <locate user group> -o udp <your mountpoint>

          For example, something like:
          sudo  ncpmount -S mywindrive -A mywindrive -U username.remote.machine.address -u $USER -g mygroup -o udp /mnt/netware/

          One word of caution: NCPmount does not map file permissions correctly, and using the -u flag simply causes the local system to regard all netware files as owned by the named user.

          You can now browse/copy files to the netware servers. It may be worth making your mounted volume accessible to you only (permissions set to 0700), as anybody logged in may be able to browse your mounted netware volumes if your home directory has read permissions to all (0755).

          The above commands must be run using sudo - so the person running them must have administrator privileges. You can make the ncpmount command 'suid', but this is a potential security hole. Only do this if you really need regular users to be able to mount Netware drives.

          chmod u+s /usr/bin/ncpmount /usr/bin/ncpumount

    4. Web Server

      • Using Bio-Linux as a webserver

        Apache2 is installed by default, but the internal firewall will only allow access to localhost. Pages placed under the folder /var/www  can be accessed using the base url:

        http://localhost/

        If you intend to run a public webserver you need to be aware of the security implications of opening up Apache to the outside world. It will be down to local IT policy whether your machine can be permitted to run any web service and who will be able to access it.

      • Setting up user html pages

        Users on your system can put static HTML and other files under a folder called public_html in their home directory. For a user with the username user1, these pages can then be accessed using the base url:

        http://localhost/~user1/

        To be viewable, the public_html directory, and any subdirectories, will have to be world executable, and the files within these directories will need to be world readable. Cgi scripts will not be executed by default. The system administrator must make alterations to the Apache configuration file to allow this. You can find detailed information on setting up Web Servers here: https://help.ubuntu.com/community/Servers

      • Running perl CGI scripts under Apache

        The default Apache setup will only run CGI scripts placed in:

        /usr/lib/cgi-bin

        Creating and running web scripts via CGI can be a fiddly process due to file permissions issues, the requirements for sending the correct headers to the server and the limitations of the HTTP protocol. Exposing your CGI on a publicly accessible site is a potential security hazard and if you intend to do this you should carefully consider your Apache setup to minimise the risk of hacking.

        Having said that, the following quick modification to the standard Apache setup will CGI-enable the public_html directories for all users. Simply edit the file /etc/apache/httpd.conf  and find the bit that looks like this:

        <Directory /home/*/public_html>
        AllowOverride FileInfo AuthConfig Limit
        Options MultiViews Indexes SymLinksIfOwnerMatch IncludesNoExec

        You want to change the last word and add a line to make it look like this:

        <Directory /home/*/public_html>
        AllowOverride FileInfo AuthConfig Limit
        Options MultiViews Indexes SymLinksIfOwnerMatch ExecCGI
        AddHandler cgi-script .cgi .sh .pl

        Then in a terminal, type:

        sudo /etc/init.d/apache2 reload

        to make Apache pick up the changes.

    5. General use

      • Working with text

        Editing text

        There are many programs on Linux that will allow you edit text files. Some of these are easy to use such as gedit and nano. Other powerful editors, with very useful features, can have a steep learning curve. Examples in this category include vim and emacs. For the new user we would recommend the graphical program gedit which can be found by searching for edit in the Dash Home. You can also run gedit directly from the command line, or use sudo gedit if you need to edit protected system files (see section 2 above).

        If you work on Bio-Linux by logging in remotely via ssh then you may not wish to run a graphical editor. In this case a good option is nano, which runs within your terminal window. Nano displays a brief list of available commands for Save, Exit, Cut, Paste... along the bottom of the window. Eg. to exit nano enter "^X" (control-X).

        Word processors such as Word and OpenOffice Writer are not plain text editors and they can introduce hidden characters and markup in your text files even when you save as a text file (with a .txt extension). Editing a text file with a word processor should be completely avoided if you plan to use the file as input to bioinformatics programs.

        Copying and pasting text

        In most graphical applications the Edit menu will contain buttons to Copy and Paste highlighted text. This works on Linux just as it does on Windows.

        You can still copy and paste text easily in applications that do not have this feature. On Linux you can assume that any highlighted text is already copied onto the clipboard. This means you usually don't need to do something special to copy text before pasting: if it's highlighted, it's already copied. Pasting the highlighted text is done by moving the mouse cursor to where you want to paste to, and then pressing both left and right mouse buttons simultaneously. If your mouse has a middle mouse button, then pressing it will also paste the text.

        Note:

        The standard keybindings in Windows for cut (ctrl-C) and paste (ctrl-V) do not apply universally under Linux as these key combinations have other features in various applications. In particular, ctrl-C is the standard keystroke for killing a running process in a terminal window!

        Or in fact you can open any file from the command line by typing:

        gnome-open filename

        This will just run whatever application would normally run if you had double-clicked the file icon.
